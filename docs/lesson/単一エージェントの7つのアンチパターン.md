# 単一エージェントの7つのアンチパターン

### 1. CLAUDE.mdを200行以上書く
指示予算（実質100〜150命令）を超えると指示無視が発生する。
- **対策**: 30〜60行に収める。詳細はADR（`docs/decisions/`）や別ファイルに分離
- **Dinekt**: ADRパターン導入済み（116行→69行）

### 2. 「〜しないこと」「〜禁止」で指示を書く
LLMは否定処理が苦手。禁止表現は遵守率が低い。
- **対策**: 「テストファイルを削除しないこと」→「テストファイルは保持すること」に書き換え
- **検証**: 否定表現を肯定に書き換えただけで遵守率向上の報告あり

### 3. タイムスタンプをシステムプロンプトに入れる
CLAUDE.mdにタイムスタンプを書くとプロンプトキャッシュが毎回無効化 → コスト5倍。
- **対策**: タイムスタンプはユーザーメッセージ側に。CLAUDE.mdには日時を書かない
- **例外**: `# currentDate` はシステムが自動注入するので問題なし

### 4. MCPを7本以上同時有効化する
ツール定義がコンテキストの10%を超えるとTool Searchが自動発動するが、それでも無駄が残る。
- **対策**: 不要なMCPは無効化。Tool Searchの自動遅延ロードを活用
- **目安**: 6本以下が理想。7本超は要検討

### 5. チェックポイントなしで30分以上のタスクを実行する
エージェントが途中で死亡すると全作業が消失。
- **対策**: 10〜15分ごとにgit commitまたはファイル保存のチェックポイント
- **実装**: エージェント定義に「15分ごとに成果をファイルに保存する」指示を含める

### 6. 長大なTODOリストを1ファイルに全部書く
回復ポイントなし・進捗不明・エージェントが方向を見失う。
- **対策**: フェーズ分割（Briefing → Planning → 実装 → Review）
- **実装**: secretary/todos/ でフェーズごとにファイル分離

### 7. 同一モデルで自己レビューする
自分が書いたコードの問題を自分で見つけられない（Self-Review Blindness）。
- **対策**: 別ロールのエージェント（批判者）にレビューさせる
- **実装**: Dinektの批判者エージェント、code-reviewerエージェント
